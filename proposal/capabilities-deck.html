The Problem

This Is a Trust Problem, Not Just a Bug List

The previous vendor left work unfinished and documentation thin. Meridian's IT team has responded rationally: they've blocked changes to a system they can't verify. R. Tanaka's operations team is working around a dashboard that doesn't fully work.

The real goal of this engagement is not to close eight tickets. It's to restore confidence — in the codebase, in the process, and in the team maintaining it. Tests aren't a deliverable. They're the proof that the work was done right.

Our Approach

Understand First. Build After.

We don't write new code until we know what we inherited. Week one is architecture review and defect audit — before any remediation begins.

Principle 1

Audit before act

Classify every defect by type before touching code. No ad-hoc fixes.

Principle 2

Tests ship with features

R3 is not a phase. Every deliverable includes its tests.

Principle 3

Leave it cleaner

The codebase we hand back should be in better shape than what we found.

R1 — Required

Reports Module Remediation

The Reports module has at least eight known defects across four filter dimensions: Time Period, Warehouse, Category, and Order Status. We will audit systematically, classify by type, and fix by category — not ad-hoc.

  • Filter wiring issues — filters not propagating correctly to the API
  • Internationalization gaps — data rendering incorrectly for non-English locales
  • Data pattern inconsistencies — some views still using legacy Options API patterns
  • Written findings document delivered before any code changes begin
6 / 13
R2 — Required

Restocking Recommendations

A new view that tells your operations team exactly what to order, for which warehouse, within a budget they set at runtime.

Input

Current stock levels

From existing inventory data already in the system.

Input

Demand forecast

From existing forecast data. Reorder threshold derived automatically.

Input

Budget ceiling

Operator-supplied at runtime. Recommendations fit within the ceiling, ranked by urgency.

Operators can accept, adjust, or dismiss individual recommendations. Filters (Warehouse, Category) apply so each warehouse can work their own queue.

R3 — Required

Automated Browser Testing

End-to-end Playwright tests — the same tool already configured in the codebase — covering the two flows Meridian IT needs most: Reports and Restocking.

Reports coverage

Filter combinations

Data display accuracy, i18n rendering, edge cases surfaced during audit.

Restocking coverage

End-to-end flow

Budget input, recommendation generation, accept/adjust/dismiss interactions.

Tests are written alongside each feature — not at the end. Deliverable: a suite Meridian IT can run via a single command, with documentation on how to extend it.

R4 — Required

Architecture Documentation

Delivered in week one. This is the reference the previous vendor didn't leave.

  • Component map — frontend views, backend routes, data files and their relationships
  • Data flow — how filters travel from the UI through the API to in-memory filtering
  • API surface — all endpoints, parameters, and response shapes
  • Known technical debt — documented honestly, so Meridian IT knows what they're holding
  • Format: interactive HTML diagram + written narrative, suitable for browser and print
